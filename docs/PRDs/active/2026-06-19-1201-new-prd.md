**PRD ID**: PRD-2026-06-14-1000

**Date**: 2026-06-14

**Status**: Implemented

**Parent Initiative**: V1.0 Release Readiness

**Target Coverage**: 100% Traceability across HTTP and Kafka boundaries, Zero Poison-Pill Event blockages.

---

## 1. Objective & Problem Statement

The ERP backend has achieved a 10/10 rating for domain isolation ($C_e = 0$) and event reliability (Inbox/Outbox). However, the distributed nature of the system introduces critical operational blind spots:

1. **No Distributed Tracing:** A single business action (e.g., Order-to-Cash) spans CRM, SCM, and FM asynchronously. There is currently no way to trace a single request ID across these network boundaries.
2. **Missing Dead Letter Queue (DLQ):** The `KafkaEventInbox` ensures exactly-once processing, but if a payload is structurally malformed (a "poison pill") or a downstream API is permanently down, the inbox retry loop will either stall or discard the event silently.
3. **Frontend Integration Friction:** The API Gateway aggregates 10 services, but there is no unified OpenAPI/Swagger specification generated from the `.cdd` contracts, making it difficult for UI/UX teams to build the frontend.
4. **Empty Multi-Tenant Databases:** Booting 10 clean PostgreSQL databases requires a deterministic data seeder to create the initial Legal Entities, Chart of Accounts, Admin Users, and Base Materials required for End-to-End testing.

## 2. Technical Scope & Architecture

### A. OpenTelemetry (OTel) Distributed Tracing

* **HTTP Middleware:** Inject OpenTelemetry middleware into the Gin routers across all 10 services to extract and inject `trace_id` headers.
* **Kafka Header Propagation:** Modify the `shared/kafka/publisher.go` and `consumer.go` to inject the `trace_id` and `span_id` into the Kafka message headers.
* **Log Correlation:** Configure the shared `logger.go` to append `[trace_id: X]` to every `log.Printf` statement for easy grep/aggregation in ELK/Loki.

### B. Dead Letter Queue (DLQ) Engine

* **Max Retries:** Update the `ReliableMessagingService` to enforce a `MAX_RETRIES` limit (e.g., 5 attempts) on the `KafkaEventInbox`.
* **DLQ Routing:** If an event exceeds the retry limit, transition its status to `FAILED_DLQ` and publish the payload to a centralized `erp.system.dlq` Kafka topic.
* **Alerting:** The DLQ acts as the primary alerting mechanism for DevOps, ensuring poison-pill messages are quarantined without blocking the rest of the event stream.

### C. Unified OpenAPI Generation & Data Seeding

* **Swagger Aggregation:** Extend the `cdd-engine` to parse all 10 `.cdd` files and output a single, unified `openapi.yaml` file hosted by the API Gateway at `/docs`.
* **The Global Seeder:** Create a `cmd/seeder/main.go` CLI script that sequentially fires API requests through the gateway to bootstrap a fresh deployment with functional test data (Users, Budgets, Warehouses).
