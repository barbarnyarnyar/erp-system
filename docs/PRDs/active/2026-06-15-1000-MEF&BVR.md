# PRD-2026-06-15-1000: Composable Micro-Frontend (MFE) & Business Value Realization

**Status**: Proposed (Draft)

**Author**: Enterprise Architecture & Business Analysis

**Parent Initiative**: V1.0 User Adoption & UI Delivery

**Target Coverage**: 100% UI/Backend Domain Alignment, Zero Frontend Monolithic Coupling.

---

## 1. Executive Summary & Problem Statement

**The Business Problem:** The backend Order-to-Cash and Procure-to-Pay loops are mathematically verified, but human operators cannot interact with Kafka topics or raw Swagger endpoints. We need a user interface. However, traditional ERP frontends degrade into "Feature Factories"—massive monolithic codebases where adding a simple button to the CRM breaks the SCM shipping screen.

**The Architectural Solution:** We will implement a **Micro-Frontend (MFE) Architecture** using Module Federation. Just as the backend is split into 10 isolated Go domains, the frontend will be split into isolated, independently deployable React/Vue applications that are stitched together at runtime inside a Host Shell.

Furthermore, to future-proof the system for autonomous AI agents, we will implement a **Backend-for-Frontend (BFF)** pattern.

---

## 2. Technical Architecture (Solution Architect Perspective)

### A. Module Federation (The Frontend Decoupling)

* **The Host Shell:** A lightweight container application responsible *only* for the global navigation bar, Auth Context (JWT passing), and routing.
* **The Remote Apps:** CRM, SCM, and FM will each have their own UI repository. They will expose their domain-specific UI components (e.g., `<SalesOrderForm />`, `<InventoryLedger />`) to the Host Shell at runtime.
* **CI/CD Autonomy:** A developer updating the HR payroll screen can merge, build via GitHub Actions/Bitbucket Pipelines, and deploy their specific MFE to production without rebuilding or redeploying the CRM or SCM frontends.

### B. Backend-for-Frontend (BFF) & Real-Time Event Fabric

* **BFF Layer:** Instead of the frontend making raw HTTP calls to 10 different microservices, a dedicated BFF service will aggregate the OpenAPI definitions and provide clean, persona-based GraphQL or REST endpoints tailored specifically for UI rendering.
* **WebSocket Ingress:** The BFF will listen to the `crm.order.confirmed` and `scm.goods.shipped` Kafka topics and push real-time WebSocket notifications to the browser, updating UI states instantly without polling.
* **Agentic Readiness:** Structuring the BFF with clean semantic actions lays the exact groundwork needed to eventually plug in autonomous AI agents for automated data entry or workflow orchestration.

## 3. Business Value Realization (Business Analyst Perspective)

To prevent technical debt from driving business decisions, the UI will be strictly partitioned by operational personas, ensuring users only load the cognitive and network weight of the modules they actually need.

### Core Workflows (V1.0 Focus)

1. **The Sales Persona (CRM UI):**
* **Scope:** View Customer profiles, generate Price Books, and hit "Confirm Order".
* **Value:** The UI strictly prevents the salesperson from viewing raw FM ledgers, reducing security surface area.


2. **The Warehouse Persona (SCM UI):**
* **Scope:** A streamlined, high-contrast dashboard showing `AllocatedQty` and `PhysicalQty`.
* **Value:** Listens to WebSocket events to pop up an alert the second a CRM order is confirmed, prompting physical allocation.


3. **The Controller Persona (FM UI):**
* **Scope:** Universal Journal ledger views and Accounts Receivable aging reports.

## 4. Scope & Implementation Checklist

### Phase 1: The UI Infrastructure (Days 1-3)

* [ ] Initialize the Host Shell application with Webpack Module Federation.
* [ ] Create a shared UI component library (Design System) for standardized buttons, tables, and typography to ensure visual consistency across independent MFEs.
* [ ] Implement the BFF (Backend-for-Frontend) API Gateway adapter.

### Phase 2: The Order-to-Cash Trinity MFEs (Days 4-8)

* [ ] **CRM Remote App:** Implement the Customer directory and Sales Order entry forms.
* [ ] **SCM Remote App:** Implement the Stock Balance table and Goods Shipment trigger buttons.
* [ ] **FM Remote App:** Implement the Ledger read-only view and Customer Credit display.

### Phase 3: Real-Time Telemetry & CI/CD (Days 9-10)

* [ ] Configure WebSockets in the BFF to push Kafka event statuses (e.g., "Outbox processing") to toast notifications in the UI.
* [ ] Set up independent CI/CD deployment pipelines for the Host, CRM, SCM, and FM frontends.

---

## 5. Definition of Done (The 10/10 Standard)

* [ ] The Frontend architecture reflects the Backend CDD boundaries; a failure in the CRM frontend codebase cannot crash the SCM warehouse screen.
* [ ] The `trace_id` generated in the browser is successfully passed through the BFF, into the Go API Gateway, and logged in the PostgreSQL Outbox.
* [ ] Updating a micro-frontend triggers a standalone CI/CD pipeline that updates the live UI without requiring a monolith redeployment.
