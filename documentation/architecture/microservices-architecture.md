# Microservices Architecture

Actual design patterns and implementation state for the ERP system's microservices.

## Service Design Principles

### Domain Boundaries
Each service maps to a business domain:

| Service | Domain | Source of Truth For |
|---------|--------|-------------------|
| **Auth** (auth-service) | User identity, roles, permissions | Users, Roles, Permissions |
| **FM** (fm-service) | Accounting & financial operations | Accounts, Journal Entries, Invoices, Payments, Budgets |
| **HR** (hr-service) | Employee lifecycle & payroll | Employees, Payroll, Timesheets, Leave, Recruitment, Training |
| **SCM** (scm-service) | Inventory & procurement | Products, Suppliers, POs, Inventory, Warehousing |
| **MFG** (m-service) | Production planning & execution | BOMs, Work Centers, Routings, Work Orders |
| **CRM** (crm-service) | Customer relationships & sales | Customers, Leads, Opportunities, Sales Orders |
| **PM** (pm-service) | Project management & billing | Portfolios, Projects, Tasks, Time Tracking |

### Autonomy — Current State

| Principle | Actual State |
|-----------|-------------|
| **Independent deployment** | ✅ Each service has its own Dockerfile and go.mod |
| **Independent data** | ⚠️ Each service owns in-memory data, **no database** |
| **Independent scaling** | ⚠️ Possible via Docker Compose, but all single-instance |
| **Independent technology** | ❌ All services use Go + Gin (consistent by design) |

## Service Architecture (Clean Architecture Lite)

### Actual Directory Structure

```
services/{service-name}/
├── cmd/
│   ├── main.go             # Entry point (most services)
│   └── server/main.go      # Alternative entry (fm-service only)
├── internal/
│   ├── api/
│   │   ├── handlers/       # HTTP handlers — one per resource
│   │   └── routes/         # Route registration (some: group handlers)
│   ├── business/
│   │   ├── domain/         # CDD-generated: models, repo interfaces, event topics
│   │   └── service/        # Business logic — one service per subdomain
│   ├── config/             # Env-based configuration
│   └── data/
│       ├── memory/         # In-memory repo implementations
│       ├── kafka/          # Producer + consumer wrappers
│       └── migrations/     # CDD-generated SQL (never executed)
├── common-utils/           # Symlink → shared/
├── go.mod / go.sum
├── Makefile (some services)
└── Dockerfile
```

### Dependency Direction

```
handlers → service → domain (interfaces)
                          ↑
                    memory/ (implementations)
                    kafka/ (implementations)
```

No DI framework — all wiring is manual in `cmd/main.go`:

```go
repo := domain.NewMemoryAccountRepo()
publisher := domain.NewKafkaPublisher(config.KafkaBrokers)
svc := service.NewAccountService(repo, publisher)
handler := handler.NewAccountHandler(svc)
routes.SetupAccountRoutes(router, handler)
```

### Key Structural Observations

- **Entry points vary**: `fm-service` uses `cmd/server/main.go`, others use `cmd/main.go`
- **No `middleware/` directory** in most services — auth, logging, CORS not implemented
- **No `pkg/` directory** — all code is internal
- **`common-utils/`** is a symlink to `shared/` but **no service uses the shared utilities**
- **Domain models** are CDD-generated with `// Code generated by CDD Engine. DO NOT EDIT.`
- **`data/migrations/`** contains SQL schemas but **no migration tool is connected**

## Inter-Service Communication

### Current State

| Pattern | Used? | Details |
|---------|-------|---------|
| **Synchronous (HTTP)** | ✅ via API Gateway | Gateway reverse-proxies requests, no direct service-to-service HTTP |
| **Asynchronous (Kafka)** | ✅ Fire-and-forget | All services produce/consume events |
| **Saga pattern** | ❌ | No distributed transaction management |
| **Circuit breaker** | ❌ | No resilience library |
| **Service discovery** | ❌ | Hardcoded URLs in gateway config |
| **Retry with backoff** | ❌ | Kafka publish is single attempt only |

### Synchronous Flow (via Gateway)

```
Client → API Gateway (:8080) → Reverse Proxy → Service (:8001-8006)
```

The gateway uses `net/http/httputil.ReverseProxy` with simple URL rewriting. No services call each other directly via HTTP.

### Asynchronous Flow (Kafka)

```
Service A → KafkaPublisher.Publish() → Kafka Broker (:9092)
                                            ↓
Service B → KafkaConsumer goroutine → ReadMessage() → switch topic → handle
```

### Kafka Integration Detail

**Producer pattern** — identical implementation across all services:

```go
// internal/data/kafka/producer.go
type KafkaPublisher struct {
    writer *kafka.Writer
}

func (p *KafkaPublisher) Publish(ctx context.Context, topic, key string, payload interface{}) error {
    bytes, _ := json.Marshal(payload)
    err := p.writer.WriteMessages(ctx, kafka.Message{
        Topic: topic,
        Key:   []byte(key),
        Value: bytes,
    })
    if err != nil {
        log.Printf("failed to publish event: %v", err)
    }
    return err  // ← always ignored by caller
}
```

**Consumer pattern** — identical goroutine per service:

```go
// internal/data/kafka/consumer.go (goroutine)
for {
    msg, err := reader.ReadMessage(ctx)
    if err != nil {
        log.Printf("error reading message: %v", err)
        time.Sleep(2 * time.Second)
        continue
    }
    switch string(msg.Topic) {
    case "fin.invoice.created":
        // handle — single goroutine, sequential
    }
}
```

## API Gateway Architecture

### Two Code Paths

| Aspect | Deployed (`cmd/main.go`) | Inactive (`internal/server/server.go`) |
|--------|------------------------|----------------------------------------|
| Proxy mechanism | `httputil.ReverseProxy` | Custom handler with header injection |
| Auth | None | JWT validation + permission middleware |
| Rate limiting | None | Per-IP sliding window |
| CORS | None | Configured middleware |
| Route matching | `r.Any("/api/v1/"+name+"/*path")` | Per-route with RBAC decorations |
| Build target | Dockerfile builds `cmd/main.go` | Must be manually built |

## Data Management

### Storage — Current State

All services use **in-memory repositories**:

```go
// internal/data/memory/account_repo.go
type MemoryAccountRepo struct {
    mu   sync.RWMutex
    data map[string]*domain.Account
}

func (r *MemoryAccountRepo) Create(ctx context.Context, a *domain.Account) error {
    r.mu.Lock()
    defer r.mu.Unlock()
    r.data[a.ID] = a
    return nil
}
```

**Characteristics**:
- `sync.RWMutex` for concurrent access
- Not persisted — **all data lost on restart**
- No queries, filtering, pagination, or sorting
- Full dataset returned on `List()` calls
- `Delete()` sets a `deleted` flag in some services, removes from map in others

### Database — Not Connected

Despite all services defining PostgreSQL connection config:
```go
type Config struct {
    DBHost     string  // default: "localhost"
    DBPort     int     // default: 5432
    DBUsername string  // default: "postgres"
    DBPassword string  // default: "postgres"
    DBDatabase string  // default: per-service DB name
}
```

**No service imports a PostgreSQL driver.** The config fields are read but never used.

SQL migration files exist at `services/*/internal/data/migrations/` (CDD-generated) but are never executed.

### Cross-Service Data Consistency

| Pattern | Implemented? | Notes |
|---------|-------------|-------|
| **Eventual consistency** | ✅ | All cross-service data via Kafka |
| **Strong consistency** | ❌ | No distributed transactions |
| **Saga / compensating transactions** | ❌ | No rollback logic |
| **Idempotency keys** | ❌ | Duplicate events may cause duplicate effects |
| **CRUD sync via events** | ✅ | FM auto-creates accounts when HR/CRM/SCM/PM events arrive |

## Resilience — Current State

| Pattern | Status | Details |
|---------|--------|---------|
| Health checks | ✅ | `GET /health` per service, returns status + service name |
| Graceful shutdown | Partial | CRM and PM handle SIGINT/SIGTERM; others don't |
| Circuit breaker | ❌ | Not implemented |
| Retry with backoff | ❌ | Kafka publish is single attempt |
| Timeout handling | ❌ | Default HTTP client (no timeout configured) |
| Dead-letter queue | ❌ | Unprocessable Kafka messages dropped |
| Rate limiting | ❌ | Code exists but not wired |
| Bulkhead | ❌ | Not implemented |

## Configuration Management

### Environment Variables

Each service loads configuration from environment variables using a `Config` struct:

```go
type Config struct {
    ServerPort   string  // default varies per service
    KafkaBrokers string  // default: "localhost:9092"
    RedisHost    string  // default: "localhost"
    RedisPort    string  // default: "6379"
    DBHost       string  // default: "localhost"
    DBPort       int     // default: 5432
    DBUsername   string  // default: "postgres"
    DBPassword   string  // default: "postgres"
    DBDatabase   string  // per-service name
    Env          string  // default: "development"
}
```

**Note**: `RedisHost/Port`, `DBHost/Port/Username/Password/Database` — values are loaded but **never used** in any service.

## Key Patterns Summary

| Concern | Pattern Used | Where |
|---------|-------------|-------|
| HTTP API | Gin routes → handlers → services → repos | All services |
| Async events | Kafka producer/consumer per service | All services |
| Storage | sync.RWMutex-protected in-memory maps | All services |
| ID generation | `prefix_` + `time.Now().UnixNano()` | All services |
| Decimal handling | `shopspring/decimal` | FM, HR, SCM, MFG, PM |
| Model generation | CDD Engine | All services (auto-generated files) |
| DI | Manual wiring in `cmd/main.go` | All services |
| Shared code | Symlinked `common-utils/` | All services (but unused) |

## Current vs: Ideal State for Microservices

| Aspect | Current | Target (for production readiness) |
|--------|---------|-----------------------------------|
| Storage | In-memory maps | PostgreSQL per service |
| Service discovery | Hardcoded URLs | Kubernetes DNS / Consul |
| Resilience | None | Circuit breakers, retries, timeouts |
| Observability | log.Printf | Structured logging, metrics, tracing |
| API auth | None | JWT + RBAC (code exists but inactive) |
| Error handling | Ad-hoc gin.H | Standardized error response format |
| Testing | 1 test file | Unit + integration + contract tests |
| CI/CD | None | Automated build + test + deploy |
| Secrets | Hardcoded defaults | Vault / K8s Secrets |
